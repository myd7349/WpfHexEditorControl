# 36 SYSTÈME DE SERVICES PARTAGÉS (SERVICE CONTAINER / DEPENDENCY INJECTION)

L’IDE doit utiliser un système de services partagés permettant aux différents composants d’accéder aux fonctionnalités communes sans créer de dépendances directes.

Ce système agit comme un conteneur central de services permettant d’enregistrer, de résoudre et de partager des services entre les composants de l’IDE.

## 36.1 Objectif du système de services

Le Service Container permet :
 
- de centraliser les services de l’IDE
- de réduire les dépendances directes entre composants
- de faciliter l’injection de dépendances
- de permettre l’extensibilité via plugins
- de simplifier la maintenance de l’architecture

Les composants doivent demander les services dont ils ont besoin plutôt que de créer directement leurs dépendances.

## 36.2 Principe de fonctionnement

Le système repose sur trois opérations principales :

### Enregistrement des services

Les services sont enregistrés dans le conteneur au démarrage de l’IDE.

Exemples :

- FileService
- EditorService
- PanelService
- PluginService
- EventBus
- TerminalService

### Résolution des services

Les composants peuvent demander un service via le conteneur.

Le conteneur retourne l’instance correspondante.

### Injection de dépendances

Les dépendances doivent être injectées via les constructeurs lorsque possible.

Cela permet de rendre les composants plus testables et plus modulaires.

## 36.3 Types de services

Le système peut inclure plusieurs catégories de services.

### Services IDE

Services centraux de l’IDE :

- gestion des fichiers
- gestion des éditeurs
- gestion des panels
- gestion des solutions
- gestion du terminal

### Services système

Services techniques :

- bus d’événements
- gestion des tâches
- gestion des logs
- gestion de la configuration

### Services plugins

Services permettant l’intégration des plugins :

- chargement des plugins
- communication avec le SDK
- gestion des extensions

## 36.4 Cycle de vie des services

Les services peuvent avoir différents cycles de vie :

| Type | Description |
|-----|-----|
| Singleton | Une seule instance pour tout l’IDE |
| Scoped | Une instance par contexte |
| Transient | Nouvelle instance à chaque demande |

La majorité des services de l’IDE doivent être **Singleton**.

## 36.5 Accès aux services par les plugins

Le SDK doit permettre aux plugins d’accéder à certains services publics de l’IDE.

Les plugins ne doivent pas accéder directement aux implémentations internes.

Le SDK doit exposer :

- des interfaces
- des services abstraits
- des points d’extension

## 36.6 Bonnes pratiques

Claude doit respecter les règles suivantes :

- ne pas instancier directement les services partagés
- utiliser l’injection de dépendances
- enregistrer les services dans le conteneur
- éviter les dépendances circulaires

Les composants doivent dépendre d’interfaces plutôt que d’implémentations concrètes.

## 36.7 Objectif

Ce système permet :

- une architecture modulaire
- une meilleure testabilité
- une extensibilité via plugins
- une maintenance facilitée
- une gestion claire des dépendances dans l’IDE

# 37 SYSTÈME DE COMMANDES GLOBAL (IDE COMMAND SYSTEM)

Le système de commandes global permet de centraliser et d’orchestrer toutes les actions exécutables dans l’IDE.  
Toutes les interactions utilisateur et automatisations passent par ce système : menus, panels, terminal, raccourcis clavier, plugins.

Claude ne doit pas créer de commandes isolées en dehors de ce système.

## 37.1 Objectif du système de commandes

Le système de commandes permet :

- un point unique pour l’exécution des actions
- l’intégration uniforme entre panels, menus et terminal
- l’extensibilité via plugins
- le support des raccourcis clavier et contextuels
- l’automatisation et le scripting

## 37.2 Structure des commandes

Chaque commande doit contenir :

- un **identifiant unique** (string ou enum)
- un **handler** (méthode ou service qui exécute l’action)
- des **paramètres** optionnels
- des **conditions d’activation** (peut ou non être exécutée selon le contexte)
- une **catégorie** (menu, terminal, panel, plugin)

Exemple de structure :

```
Command
├─ Id: "OpenFile"
├─ Handler: FileService.Open()
├─ Parameters: FilePath
├─ CanExecute: CurrentSolutionLoaded
├─ Category: IDE/Menu
```

## 37.3 Intégration avec l’IDE

Le système de commandes doit être utilisé par :

- les menus et sous-menus
- les barres d’outils
- le terminal interne
- les panels
- les plugins via le SDK

Claude doit s’assurer que toutes les actions passent par le **CommandBus central**.

## 37.4 CommandBus central

Le CommandBus central :

- reçoit toutes les commandes
- valide le contexte d’exécution
- exécute le handler associé
- notifie les composants si nécessaire
- log les actions pour débogage ou scripts

## 37.5 Extensibilité pour les plugins

Le SDK doit permettre aux plugins :

- de déclarer de nouvelles commandes
- de fournir des handlers
- de définir les paramètres et conditions d’exécution
- de s’intégrer dans les menus ou le terminal

Les plugins ne doivent pas contourner le CommandBus central.

## 37.6 Commandes contextuelles

Les commandes doivent pouvoir :

- être activées seulement dans certains contextes (fichier ouvert, panel actif, solution chargée)
- être accessibles via menu contextuel
- être accessibles via raccourcis clavier configurables
- être appelées depuis le terminal ou les scripts

## 37.7 Bonnes pratiques

Claude doit respecter :

- l’unicité des identifiants de commandes
- l’utilisation du CommandBus central
- la séparation des handlers et de l’UI
- la documentation des commandes pour le SDK
- l’implémentation des conditions de contexte pour éviter les erreurs

## 37.8 Objectif

Ce système permet :

- une architecture cohérente pour toutes les actions de l’IDE
- l’extensibilité via plugins
- la compatibilité avec le terminal et le scripting
- une interface utilisateur prévisible et fiable
- un contrôle total sur l’exécution des actions

# 38 SYSTÈME DE RACCOURCIS CLAVIER ET BINDINGS DE COMMANDES

Pour un IDE moderne et extensible, toutes les commandes doivent pouvoir être déclenchées via des raccourcis clavier configurables.  
Le système de raccourcis doit s’intégrer avec le **CommandBus global** (#37) pour assurer la cohérence et l’extensibilité.

Claude ne doit pas créer de raccourcis indépendants hors de ce système.

## 38.1 Objectif

Le système de raccourcis permet :

- d’exécuter toutes les commandes de manière rapide
- d’améliorer l’ergonomie et la productivité
- de supporter la personnalisation par l’utilisateur
- de permettre aux plugins d’ajouter leurs propres raccourcis

## 38.2 Structure des bindings

Chaque binding doit contenir :

- un **identifiant unique** lié à la commande
- la **combinaison de touches** (ex. Ctrl+O)
- une **catégorie/context** (menu, panel, terminal, global)
- des **conditions d’activation** (ex. fichier ouvert, panel actif)

Exemple :

```
KeyBinding
├─ CommandId: "OpenFile"
├─ KeyGesture: "Ctrl+O"
├─ Context: Global
├─ CanExecute: CurrentSolutionLoaded
```

## 38.3 Gestion centralisée

Un service central de raccourcis doit :

- gérer l’association touche → commande
- vérifier les conditions de contexte avant exécution
- intercepter les touches même dans les panels ou éditeurs
- permettre l’ajout, la suppression ou la modification dynamique des bindings

## 38.4 Extensibilité pour les plugins

Les plugins doivent pouvoir :

- déclarer de nouvelles commandes avec des raccourcis par défaut
- enregistrer leurs bindings dans le service central
- permettre aux utilisateurs de reconfigurer les raccourcis via l’UI

Claude doit s’assurer que les raccourcis des plugins respectent le système central et ne créent pas de conflits non gérés.

## 38.5 Gestion des conflits

Le système doit détecter :

- les combinaisons de touches en double
- les conflits de contextes
- les bindings invalides

En cas de conflit, le système peut :

- prioriser les bindings de l’IDE par défaut
- notifier l’utilisateur pour les plugins
- permettre une reconfiguration manuelle

## 38.6 Configuration utilisateur

Le système doit permettre :

- de visualiser tous les raccourcis
- de modifier les raccourcis existants
- de restaurer les raccourcis par défaut
- d’exporter / importer des configurations

## 38.7 Bonnes pratiques

Claude doit respecter :

- l’utilisation du CommandBus global pour l’exécution
- la séparation entre logique de commande et bindings
- la documentation des bindings pour le SDK et les plugins
- la gestion centralisée des conflits

## 38.8 Objectif

Ce système permet :

- une expérience utilisateur cohérente et productive
- une intégration complète avec toutes les commandes de l’IDE
- l’extensibilité via plugins
- une personnalisation totale des raccourcis
- un contrôle clair sur l’exécution et le contexte des commandes

# 39 CONFIGURATION ET PERSISTANCE DES PRÉFÉRENCES UTILISATEUR

Pour un IDE moderne, toutes les préférences utilisateur (UI, raccourcis, panels, thèmes, plugins, paramètres divers) doivent être centralisées et persistées afin d’assurer une expérience cohérente entre les sessions.

Claude ne doit pas créer de configurations dispersées ou non persistantes.

## 39.1 Objectif

Le système de configuration doit permettre :

- de centraliser tous les paramètres de l’IDE
- de sauvegarder les préférences entre les sessions
- de permettre la personnalisation complète par l’utilisateur
- de fournir une API aux plugins pour stocker leurs paramètres
- de faciliter l’export/import de configurations

## 39.2 Architecture recommandée

- **ConfigurationManager** : service central gérant les préférences
- **StorageProvider** : interface pour la persistance (fichier JSON, base de données locale, registre, etc.)
- **Schema de configuration** : chaque module/plugin expose ses paramètres de manière structurée

Structure simplifiée :

```
ConfigurationManager
├─ Load()
├─ Save()
├─ GetSection(name)
├─ SetValue(section, key, value)
├─ RegisterPluginConfig(pluginId, schema)
```

## 39.3 Types de préférences

1. **UI / Panels** : position des panels, thèmes, polices
2. **Raccourcis clavier** : bindings configurés (#38)
3. **Terminal** : history, préférences d’affichage
4. **Plugins** : paramètres spécifiques exposés via le SDK
5. **IDE** : chemins par défaut, options de build, préférences générales

## 39.4 Persistance

Le système doit :

- charger les paramètres au démarrage
- sauvegarder automatiquement ou sur demande
- gérer les migrations de version
- permettre l’export/import de configuration pour partage ou backup

## 39.5 Extensibilité pour les plugins

- Les plugins doivent pouvoir déclarer leurs sections de configuration
- Le SDK fournit l’accès au ConfigurationManager
- Les plugins peuvent sauvegarder et charger leurs préférences sans toucher aux fichiers de l’IDE directement

## 39.6 Bonnes pratiques

Claude doit respecter :

- centraliser toutes les préférences via le ConfigurationManager
- éviter les fichiers ou variables locales dispersées
- documenter les clés de configuration pour le SDK
- gérer les valeurs par défaut et les migrations de versions

## 39.7 Objectif

Ce système permet :

- une expérience utilisateur cohérente
- la personnalisation complète de l’IDE
- la maintenance facile des paramètres
- l’extensibilité pour les plugins
- la sauvegarde et restauration des préférences entre sessions
