
# 77 WORKSPACE / ENVIRONMENT SYSTEM

## Objectif

Implanter un système de Workspace permettant de gérer plusieurs environnements de travail complets dans l'IDE.

Un Workspace représente un contexte de travail contenant :

- projets / solutions ouverts
- layout des panels
- plugins actifs
- configuration terminal
- paramètres des éditeurs (CodeEditor, HexEditor, JsonEditor)
- thèmes
- raccourcis clavier
- overrides d’options spécifiques

Le changement de Workspace reconfigure automatiquement l’environnement de l’IDE.

---

## Structure générale

Le Workspace agit comme une couche entre :

Runtime IDE  
↓  
Workspace  
↓  
OptionsDocument (global)

Architecture :

Global Options  
→ Workspace Overrides  
→ Runtime Configuration

Le Workspace peut remplacer certaines options globales sans modifier le OptionsDocument principal.

---

## Gestion des Workspaces

La gestion principale se fait via le menu File :

File  
 ├── New Workspace  
 ├── Open Workspace  
 ├── Save Workspace  
 ├── Save Workspace As  
 ├── Switch Workspace  
 └── Close Workspace  

Un Workspace est enregistré sous forme de fichier :

*.whidews

Exemples :

workspace_reverse_engineering.whidews  
workspace_dev_csharp.whidews  
workspace_data_analysis.whidews  

---

## Contenu d’un Workspace

Structure recommandée :

workspace.whidews
│
├── workspace.json  
├── layout.json  
├── plugins.json  
├── openfiles.json  
├── terminal.json  
└── settings_override.json  

Description :

workspace.json  
    métadonnées du workspace

layout.json  
    disposition des panels

plugins.json  
    plugins requis / optionnels / désactivés

openfiles.json  
    fichiers ouverts au moment de la sauvegarde

terminal.json  
    configuration terminal

settings_override.json  
    override des options globales

---

## Activation automatique des plugins

Un Workspace peut définir :

RequiredPlugins  
OptionalPlugins  
DisabledPlugins  

Lors du chargement :

PluginManager active ou désactive automatiquement les plugins requis.

Cela permet d’optimiser les performances et de charger seulement les plugins nécessaires.

# 78 COMMAND SYSTEM (CENTRAL COMMAND BUS)

## Objectif

Implanter un système centralisé de commandes permettant à toutes les composantes de l’IDE d’exécuter des actions de manière uniforme.

Le Command System devient la couche standard pour déclencher des actions dans l’IDE.

Toutes les actions importantes du système passent par ce mécanisme :

- menu
- raccourcis clavier
- terminal
- scripts
- plugins
- future intégration IA

Cela garantit une architecture cohérente et extensible.

---

## Principe

Chaque action de l’IDE est représentée par une Command.

Exemples :

File.OpenFile  
File.CloseFile  
Workspace.Switch  
Plugin.Enable  
Terminal.Execute  
Solution.Build  

Les commandes peuvent être appelées par différents systèmes.

---

## Sources d’exécution des commandes

Une commande peut être déclenchée par :

Menu UI  
Keyboard Shortcut  
Terminal Command  
Script  
Plugin  
Internal IDE Logic  
Future AI assistant

Architecture :

UI / Terminal / Script / Plugin
            ↓
        CommandBus
            ↓
      Command Handlers
            ↓
        IDE Services

---

## Structure d’une commande

Chaque commande possède :

CommandId  
Description  
Category  
Handler  
Parameters  

Exemple :

CommandId : File.OpenFile  
Category : File  
Handler : FileService.OpenFile

---

## Exemple de définition

Command :

File.OpenFile

Paramètres :

path

Execution :

CommandBus.Execute("File.OpenFile", { path })

---

## Command Registry

Un registre central maintient la liste de toutes les commandes disponibles.

CommandRegistry

Responsabilités :

- enregistrer les commandes
- exposer les commandes disponibles
- permettre aux plugins d’ajouter leurs commandes

---

## Command Handlers

Chaque commande possède un Handler responsable de son exécution.

Exemple :

FileOpenCommandHandler

Responsabilités :

- validation paramètres
- exécution action
- retour résultat

---

## Intégration avec les plugins

Les plugins peuvent enregistrer leurs propres commandes.

Exemple :

HexEditor.GoToOffset  
JsonEditor.FormatDocument  
BinaryAnalyzer.ScanFile

Lors du chargement du plugin :

le plugin enregistre ses commandes dans le CommandRegistry.

---

## Intégration avec le terminal

Le terminal peut appeler les commandes.

Exemple :

command File.OpenFile path="test.bin"

command Workspace.Switch name="ReverseEngineering"

Cela transforme le terminal en interface de contrôle complète de l’IDE.

---

## Intégration avec les raccourcis clavier

Keybindings :

Ctrl + O → File.OpenFile  
Ctrl + Shift + W → Workspace.Switch  

Les raccourcis déclenchent simplement des commandes.

---

## Intégration avec les scripts

Le futur langage de script de l’IDE pourra appeler les commandes.

Exemple :

openFile("test.bin")

workspaceSwitch("ReverseEngineering")

Les scripts deviennent simplement une couche sur le Command System.

---

## Module architecture

Nouveau module :

WpfHexEditor.Commands

Composants principaux :

CommandBus  
CommandRegistry  
CommandHandler  
CommandContext  
CommandResult  

---

## Interaction avec les autres modules

CommandBus
    ├── PluginManager
    ├── WorkspaceManager
    ├── SolutionManager
    ├── TerminalManager
    └── EditorServices

Les modules enregistrent leurs commandes lors de leur initialisation.

# 79 SCRIPTING SYSTEM

Le scripting permet aux utilisateurs, plugins et outils externes d'exécuter des séquences de commandes pour automatiser des tâches.

Le système de script repose directement sur le Command System (#78).

Chaque instruction du script appelle une commande du CommandBus.

---

## Cas d’utilisation

Automatisation de tâches répétitives.

Exemples :

ouvrir un fichier  
analyser un binaire  
formater un document  
exécuter un build  
lancer un plugin  

---

## Exemple de script

script :

openFile "test.bin"

hex.gotoOffset 0x400

binary.scan

saveFile

---

## Execution depuis le terminal

Le terminal peut exécuter un script.

Exemple :

runScript analyze_binary.whiscript

---

Les scripts utilisent l’extension :

*.whiscript

Exemples :

analyze_binary.whiscript  
format_json.whiscript  
build_project.whiscript  

---

## Structure d’un script

Un script est une séquence de commandes.

Exemple :

# 80 EVENT SYSTEM (IDE EVENT BUS)

## Objectif

Implanter un système d’événements centralisé permettant aux modules de l’IDE et aux plugins de communiquer entre eux de manière découplée.

Le Event System permet aux composants du système de publier et d’écouter des événements sans dépendance directe entre les modules.

Ce mécanisme est essentiel pour :

- l’architecture plugin
- l’extensibilité du système
- la communication entre modules
- l’automatisation avancée
- l’intégration de scripts et d’IA

---

## Principe

Les modules et plugins peuvent :

Publier des événements  
S’abonner à des événements  

Architecture :

Module / Plugin
      ↓
   EventBus
      ↓
Event Subscribers

---

## Exemple d’événements

FileOpened  
FileClosed  
WorkspaceChanged  
PluginLoaded  
PluginUnloaded  
DocumentSaved  
EditorSelectionChanged  
TerminalCommandExecuted  

---

## Exemple de flux

Utilisateur ouvre un fichier :

FileService
    ↓
Publish Event : FileOpened
    ↓
EventBus
    ↓
Subscribers

Plugins ou modules abonnés reçoivent l’événement.

---

## Exemple concret

Plugin BinaryAnalyzer

écoute :

FileOpened

Si le fichier est un binaire :

→ déclenche une analyse automatique.

---

## Structure d’un événement

Un événement contient :

EventId  
Source  
Timestamp  
Payload  

Exemple :

EventId : FileOpened

Payload :

filePath  
fileType  

---

## Publication d’événements

Exemple :

EventBus.Publish(
    "FileOpened",
    {
        path : "test.bin"
    }
)

---

## Souscription aux événements

Exemple :

EventBus.Subscribe(
    "FileOpened",
    HandleFileOpened
)

---

## Intégration avec les plugins

Les plugins peuvent :

publier des événements  
écouter des événements  

Exemples :

HexEditor :

HexSelectionChanged  
HexOffsetChanged  

JsonEditor :

JsonFormatted  
JsonValidationError  

BinaryAnalyzer :

BinaryScanCompleted  

---

## Intégration avec le Scripting System

Les scripts peuvent réagir aux événements.

Exemple :

on FileOpened

runScript analyze_binary.whiscript

Cela permet de créer des automatisations basées sur les actions de l’utilisateur.

---

## Intégration avec le Command System

Les commandes peuvent publier des événements.

Exemple :

Command :

File.OpenFile

après exécution :

EventBus.Publish("FileOpened")

---

## Module architecture

Nouveau module :

WpfHexEditor.Events

Composants principaux :

EventBus  
EventRegistry

EventContext  

---

## Interaction avec les autres modules

EventBus
    ├── PluginManager
    ├── WorkspaceManager
    ├── CommandBus

    └── TerminalManager

---

## Avantages

Architecture découplée

Les modules n’ont pas besoin de dépendre directement les uns des autres.

Extensibilité

Les plugins peuvent réagir aux actions de l’IDE.

Automatisation

Les scripts peuvent être déclenchés par événements.

Flexibilité

Les nouveaux modules peuvent facilement s’intégrer au système.

---

## Résultat

Le Event System devient la colonne vertébrale de communication de l’IDE.

- communication inter-modules

- automatisation basée sur événements
- intégration plugins propre

# 81 PLUGIN SANDBOX & ISOLATION SYSTEM

## Objectif

Implanter un système d’isolation des plugins afin d'assurer la stabilité, la sécurité et la fiabilité de l’IDE.

Les plugins doivent être exécutés dans un environnement contrôlé (sandbox) afin d’éviter :

- les crashes de l’IDE
- les accès non autorisés
- les conflits entre plugins
- les problèmes de dépendances

Le Plugin Sandbox System permet de séparer l'exécution des plugins du cœur de l’IDE.

---

## Principe

Chaque plugin est chargé dans un contexte isolé.

Architecture :

IDE Core  
    ↓  
Plugin Manager  
    ↓  
Plugin Sandbox  
    ↓  
Plugin Instance

Les communications passent par les services publics de l’IDE.

Les plugins n'accèdent jamais directement au cœur du système.

---

## Isolation des dépendances

Chaque plugin peut posséder ses propres dépendances.

Structure :

plugins
 ├── PluginA
 │     ├── plugin.manifest
 │     ├── plugin.dll
 │     └── dependencies
 │
 └── PluginB
       ├── plugin.manifest
       ├── plugin.dll
       └── dependencies

Cela évite les conflits de versions entre plugins.

---

## Chargement isolé

Les plugins sont chargés dans un contexte séparé.

Exemple :

PluginLoadContext

Chaque plugin possède :

- son AssemblyLoadContext
- ses dépendances isolées
- son cycle de vie

---

## Cycle de vie d’un plugin

PluginManager

Load  
Initialize  
Activate  
Deactivate  
Unload  

Le sandbox garantit qu’un plugin peut être retiré sans redémarrer l’IDE.

---

## Permissions des plugins

Le sandbox peut définir des permissions.

Exemples :

FileAccess  
NetworkAccess  
TerminalAccess  
WorkspaceAccess  
EditorAccess  

Un plugin déclare ses permissions dans son manifest.

---

## Exemple de manifest

plugin.manifest

Version

Exemple :

RequiredPermissions

FileAccess  
EditorAccess  

---

## Communication avec l’IDE

Les plugins communiquent avec l’IDE via :

CommandBus  
EventBus  
IDE Services  

Architecture :

Plugin
   ↓
IDE API
   ↓
CommandBus / EventBus

---

## Détection de crash plugin

Si un plugin provoque une exception non gérée :

PluginManager peut :

- désactiver le plugin
- enregistrer le crash
- notifier l’utilisateur

Cela évite qu’un plugin fasse planter l’IDE.

---

## Monitoring des plugins

Le système peut surveiller :

- consommation mémoire
- temps d’exécution
- erreurs

Cela permet de diagnostiquer les plugins problématiques.

---

## Intégration avec le Marketplace

Les plugins provenant du marketplace peuvent être :

- vérifiés
- signés
- isolés dans leur sandbox

Le sandbox devient un élément clé pour la sécurité du marketplace.

## Avantages

Stabilité

Un plugin défectueux ne fait pas planter l’IDE.

Sécurité

Les accès sont contrôlés.

Isolation

Les dépendances des plugins ne se mélangent pas.

Marketplace prêt

Le système permet d’accepter des plugins tiers en toute sécurité.

---

## Résultat

Le Plugin Sandbox System transforme l’IDE en véritable plateforme extensible sécurisée.

# 82 IDE SERVICE ARCHITECTURE (SERVICE REGISTRY & DEPENDENCY INJECTION)

## Objectif

Implanter un système de **Service Registry** et de **Dependency Injection** pour centraliser et gérer tous les services de l’IDE.

Cela permet aux modules et plugins de :

- accéder aux services de l’IDE de manière propre
- éviter les dépendances directes
- améliorer la testabilité
- simplifier l’extension et l’injection de nouvelles fonctionnalités

---

## Principe

Tout service de l’IDE est enregistré dans un **Service Registry**.

Exemples de services :

- FileService  
- WorkspaceManager  
- CommandBus  
- EventBus  
- PluginManager  
- TerminalManager  
- EditorServices  

Les modules et plugins obtiennent les services via l’injection de dépendances plutôt qu’en les instanciant eux-mêmes.

---

## Architecture

ServiceRegistry
├── Register<Service>(instance)
├── Resolve<Service>()
└── Unregister<Service>()


Chaque module ou plugin demande un service :

```csharp
var fileService = ServiceRegistry.Resolve<IFileService>();
fileService.OpenFile("test.bin");

Avantages

Découplage

Les modules ne dépendent pas directement les uns des autres.

Le Plugin Sandbox n’a pas besoin de connaître l’implémentation interne des services.

Extensibilité

Ajouter un nouveau service est simple.

Les plugins peuvent s’abonner ou consommer un service existant.

Testabilité

Permet de remplacer facilement un service par un mock pour les tests.

Centralisation

Tous les services sont accessibles depuis un point unique.

Exemple d’intégration

Module CodeEditor :

dépend de TerminalService et WorkspaceService

ces services sont injectés à la création du module

pas besoin de connaître la logique interne du terminal

Plugin HexEditor :

récupère EventBus et CommandBus via ServiceRegistry

déclenche des commandes et écoute des événements

fonctionne dans le Plugin Sandbox

Services recommandés à centraliser

FileService

WorkspaceManager

CommandBus

EventBus

PluginManager

TerminalManager

EditorServices

LocalizationManager

ThemeManager

OptionsDocument

Injection dans les plugins

Lors de l’initialisation du plugin :

plugin.Initialize(ServiceRegistry);

Le plugin peut ensuite résoudre tous les services dont il a besoin :

var commandBus = ServiceRegistry.Resolve<ICommandBus>();
var workspace = ServiceRegistry.Resolve<IWorkspaceManager>();
Module architecture

Nouveau module :

WpfHexEditor.ServiceRegistry

Composants principaux :

ServiceRegistry

IService interface

DependencyResolver

ServiceProvider

Interaction avec les autres modules

ServiceRegistry
├── PluginManager
├── WorkspaceManager
├── CommandBus
├── EventBus
├── TerminalManager
└── EditorServices

Tous les modules et plugins interagissent via les services injectés.
