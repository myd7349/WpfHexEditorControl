# WpfHexEditor — Enterprise Plugin System (SDK Public)
## Microsoft-Level Architecture Plan
### Namespace racine : `WpfHexEditor`

---

# 1. OBJECTIFS STRATÉGIQUES

Créer un système de plugins :

- 100% Open-Source (aucun plugin payant)
- Extensible via SDK public
- Charge / Décharge à chaud (live)
- Isolation complète (aucun crash plugin ne doit affecter l’IDE)
- Monitoring avancé (CPU / Mémoire / Freeze detection)
- Gestion des permissions granulaire
- Intégration profonde UI (menu, toolbar, panel, document, statusbar, Terminal)
- Focus-aware (panel/document actif)
- Thème obligatoire (respect UI IDE) sauf si plugin d'extention du TerminalPanel (doit être défini dans le manifest sans quo "isTerminalOnly = True")
- Sérialisation état plugins
- Détection plugins lents (VS-like suggestion de désactivation)
- Compatibilité ascendante pour les plugins tiers afin que des versions anciennes restent compatibles avec les nouvelles versions de l’IDE.

## 2.2 WpfHexEditor.SDK

**Objectif :** Fournir un ensemble complet d’interfaces, classes et services pour permettre aux développeurs de créer des plugins pour WpfHexEditorIDE, en garantissant isolation, extensibilité et compatibilité avec le core de l’IDE.

### Contenu et responsabilités du SDK

- **Interfaces de plugin** :
  - `IWpfHexEditorPlugin` : interface principale à implémenter par tout plugin.
  - `IPluginState` : gestion de la sérialisation / désérialisation de l’état plugin.
  - `IPluginDiagnostics` : diagnostic et monitoring (CPU, mémoire, temps d’exécution, responsiveness).

- **Services exposés aux plugins** via `IIDEHostContext` :
  - `ISolutionExplorerService` : navigation et accès aux fichiers/projets.
  - `IHexEditorService` : accès au contenu binaire/hex, sélection active, offset.
  - `ICodeEditorService` : accès à l’éditeur de texte/code.
  - `IOutputService` : log et sortie console dédiée plugin.
  - `IParsedFieldService` : services d’analyse et parsing de champs structurés.
  - `IErrorPanelService` : affichage d’erreurs spécifiques au plugin.
  - `IFocusContextService` : suivi panel/document actif, focus-aware.
  - `IThemeService` : intégration Dark/Light mode et couleurs dynamiques.
  - `IPermissionService` : gestion granulaire des permissions plugin.

- **UI Registry et intégration visuelle** :
  - `IUIRegistry` : API pour enregistrer menu, toolbar, panel, document tab ou popup menu.
  - Support de **dockable panels**, **document tabs**, et intégration toolbar/menu.

- **Gestion cycle de vie plugin** :
  - Méthodes `InitializeAsync` / `ShutdownAsync` pour load/unload à chaud.
  - Hot reload supporté via `AssemblyLoadContext` isolé.
  - Sécurité et crash-resilience assurées par PluginHost et éventuellement PluginSandbox.

- **Monitoring et diagnostics** :
  - Plugin diagnostics intégré : CPU, mémoire, temps moyen exécution, freeze detection.
  - Alerts pour plugins lents ou non-responsives.

- **Sérialisation et état** :
  - Sauvegarde automatique dans `%AppData%/WpfHexEditor/Plugins/{PluginId}.json`.
  - Permet reprise de l’état au démarrage de l’IDE.

- **Extensibilité et compatibilité** :
  - Permet d’ajouter de nouveaux types de panels, document tools, parseurs, analyzers.
  - Compatible avec tous les plugins officiels et tiers.
  - SDK versionné pour assurer compatibilité ascendante.

  **Plugin Inter-dependance**
  - Les transitions entre plugins (Data Inspector → File Statistics → Pattern Analysis) sont utiles, mais on doit définir un modèle d’événements ou bus interne pour éviter des dépendances trop fortes.

**Résumé :**  
`WpfHexEditor.SDK` agit comme **couche intermédiaire officielle entre le Core de l’IDE et les plugins**, garantissant isolation, sécurité, monitoring, UI intégrée et cycle de vie complet pour chaque plugin, tout en offrant une base solide pour les développeurs externes.

## 2.3 WpfHexEditor.PluginHost

**Objectif :** Servir de gestionnaire central des plugins dans WpfHexEditorIDE, responsable du chargement, de l’exécution, du monitoring et de l’isolation standard des plugins.

### Contenu et responsabilités

- **Chargement et gestion des plugins** :
  - Découverte automatique des plugins installés.
  - Chargement via `AssemblyLoadContext` isolé pour chaque plugin.
  - Gestion du cycle de vie : `InitializeAsync` / `ShutdownAsync`.
  - Support du **hot-load / hot-unload** sans redémarrage IDE.

- **Monitoring et diagnostics** :
  - Surveillance CPU, mémoire et responsiveness de chaque plugin.
  - Détection de plugins lents ou bloqués.
  - Logs et reporting vers `IOutputService`.

- **Sécurité et stabilité** :
  - Isolation partielle pour éviter que les exceptions d’un plugin crashent l’IDE.
  - Gestion des plugins Faulted et notifications utilisateur (VS-like).
  - Interaction avec `IPermissionService` pour respecter les droits déclarés.

- **UI et intégration** :
  - Coordination avec `IUIRegistry` pour enregistrer menu, toolbar, panels.
  - Propagation des événements focus-aware via `IFocusContextService`.

**Résumé :**  
`PluginHost` est le cœur opérationnel qui orchestre les plugins dans l’IDE, garantissant stabilité, monitoring, UI intégrée et cycle de vie complet pour chaque plugin.

---

## 2.4 WpfHexEditor.PluginSandbox (option extrême isolation)

**Objectif :** Fournir une isolation maximale pour les plugins à risque ou expérimentaux, en exécutant chaque plugin dans un processus séparé.

### Contenu et responsabilités

- **Exécution en processus séparé** :
  - Chaque plugin peut tourner dans `PluginSandbox.exe`.
  - Communication sécurisée avec le PluginHost via **Named Pipes** ou **gRPC**.
  - Crash du processus plugin totalement isolé de l’IDE.

- **Monitoring OS-level** :
  - Surveillance CPU/mémoire indépendante via OS.
  - Possibilité de tuer immédiatement le processus si le plugin est non-responsif.
  - Prévoir une fréquence de collecte configurable (dans Plugin manager) pour CPU/mémoire pour ne pas surcharger l’IDE.
  - Pour les plugins très lourds, envisager un throttling automatique du rafraîchissement graphique.

- **Sécurité et permissions** :
  - Isolation complète pour protéger l’IDE et les données utilisateur.
  - Vérification stricte des permissions avant toute opération.

- **Scénarios d’usage** :
  - Plugins expérimentaux.
  - Plugins tiers non vérifiés.
  - Tests de performance ou stress sans risque.

**Résumé :**  
`PluginSandbox` est la solution ultime d’isolation, garantissant que les plugins critiques ou non fiables n’affectent jamais la stabilité de WpfHexEditorIDE.

## 3.3 Plugin Manifest (Obligatoire)

Chaque plugin doit fournir un fichier manifest décrivant son identité, sa compatibilité et ses permissions.

Chaque plugin peut aussi inclure des métadonnées avancées permettant :

- Vérification d’intégrité (hash)
- Signature numérique
- Validation éditeur
- Intégration future Marketplace
- Trust model (éditeur approuvé)

Élements à considerer:
- Prevoir un champ dependencies pour indiquer si un plugin nécessite d’autres plugins actifs.
- Prevoir un champ loadPriority pour gérer l’ordre de chargement (doit tenir compte des dependencies)
- intégrer un timestamping pour sécurité à long terme.
- Préciser un ID unique (IUIRegistry) pour chaque élément UI pour éviter collision lors du hot-load.

Nom du fichier :
WpfHexEditor.plugin.json

- Identifier formellement le plugin
- Déclarer la version et la compatibilité SDK
- Définir les permissions requises
- Déclarer le mode d’isolation (InProcess / Sandbox)

### Champs requis

- id : identifiant unique global
- name : nom affiché
- version : version plugin (SemVer obligatoire)
- sdkVersion : version minimale SDK requise
- entryPoint : classe implémentant IWpfHexEditorPlugin
- isolationMode : InProcess | Sandbox
- permissions : capacités déclarées

### Validation au chargement

Au moment du load :

1. Lire le manifest
2. Vérifier compatibilité version IDE
3. Vérifier compatibilité SDK
4. Vérifier permissions
5. Choisir mode isolation
6. Charger assembly
7. Instancier plugin

Si échec → plugin marqué "Incompatible" ou "Blocked"

### Rôle du PluginHost

- Parser le manifest
- Appliquer les permissions déclarées
- Sélectionner isolation mode
- Refuser chargement si incompatibilité

### Versioning Strategy

- SemVer obligatoire
- Breaking changes SDK → bump major
- PluginHost doit supporter compat ascendante si possible
- Possibilité future : système de migration automatique

## 3.4 Plugin Manifest — Build & Packaging Lifecycle

Le système de plugin repose sur un manifest généré automatiquement à la compilation, puis finalisé lors du packaging en `.whxplugin`.

L’objectif est de séparer clairement :

- La responsabilité du développeur
- La génération automatique au build
- La sécurisation lors de la distribution
- La validation au runtime

---

### 3.4.1 Manifest de Build (Generated at Compilation)

Lors de la compilation d’un plugin basé sur `WpfHexEditor.SDK`, un fichier :

manifest.json

est généré automatiquement via un target MSBuild.

Ce manifest contient uniquement les métadonnées déclaratives :

- id
- name
- version
- author
- sdkVersion
- minIDEVersion
- entryPoint
- permissions
- isolationMode
- description

Ce manifest est suffisant pour :

- Chargement local en mode développement
- Tests internes
- Debug
- Validation SDK

Il ne contient PAS :

- Hash de l’assembly
- Signature numérique
- Métadonnées marketplace finales
- Validation trustedPublisher

Ce manifest est appelé :

Build Manifest

---

### 3.4.2 Manifest de Distribution (Finalized During Packaging)

Lors du packaging via l’outil dédié (ex: whxpack), le Build Manifest est enrichi.

Le PackagingTool :

1. Charge le manifest généré
2. Calcule le SHA-256 du DLL compilé
3. Injecte le hash dans la section assembly
4. Injecte la taille du fichier
5. Optionnellement signe le package
6. Ajoute les métadonnées marketplace si présentes
7. Génère le fichier `.whix`

À ce stade, le manifest devient :

Distribution Manifest

Il contient désormais :

- Hash vérifiable
- Signature optionnelle
- Données marketplace
- trustedPublisher
- Intégrité validable

---

### 3.4.3 Séparation des Responsabilités

Le système respecte une séparation stricte :

WpfHexEditor.SDK  
→ Définit les interfaces et contrats  

MSBuild Target  
→ Génère le Build Manifest  

PackagingTool  
→ Finalise le manifest et crée `.whxplugin`  

WpfHexEditor.PluginHost  
→ Vérifie intégrité et charge le plugin  

Cette séparation garantit :

- Aucun couplage entre runtime et packaging
- Évolution indépendante des outils
- Sécurité renforcée
- Compatibilité CI/CD
- Scalabilité marketplace future

---

### 3.4.4 Lifecycle Complet

Développement  
→ dotnet build  
→ Génération automatique manifest.json  
→ WhixPack build  
→ Injection hash + signature  
→ Génération Plugin.whix
→ Installation via PluginHost  
→ Vérification sécurité  
→ Chargement contrôlé

---

### 3.4.5 Objectif Architectural

Le système est conçu pour permettre :

- Un développement simple
- Une distribution sécurisée
- Un futur marketplace officiel
- Un modèle enterprise avec trustedPublisher
- Une validation d’intégrité systématique

Le manifest est donc un artefact évolutif, passant de :

Build Context à Distribution Context

# 13a. Transition des modules actuel vers la nouvelles architetures plugins

## Objectif
- Transformer chaque module en plugin indépendant.
- Chaque plugin dispose de sa **propre catégorie** dans l’interface de gestion des plugins.
- Permettre des **transitions de données ou actions entre plugins**, comme pour le Panel Monitoring (Barchart).

## Modules à transformer en plugins
- **Data Inspector (Inspecteur de données)** → plugins pour visualiser et inspecter les données du fichier.
- **Structure Overlay (Superposition de structure)** → plugins pour superposer des informations structurelles sur le fichier.
- **File Statistics (Statistiques du fichier)** → plugins générant statistiques et métriques.
- **Pattern Analysis (Analyse de motifs)** → plugins pour détecter motifs et répétitions dans le fichier.
- **Format Info (Informations sur le format)** → plugins pour identifier et détailler le format du fichier.
- **File Comparison (Comparaison de fichiers)** → plugins pour comparer fichiers ou blocs de données.
- **Archive Structure (Structure de l’archive)** → plugins pour visualiser les archives (ZIP, TAR, etc.).
- **Custom Parser Template (Modèle de parseur personnalisé)** → plugins permettant d’ajouter des parseurs personnalisés.

## Exemple : Panel Monitoring (Barchart)
- Panel Monitoring = plugin sous **Data Inspector**
- Inclut barchart, toolbar, alertes
- Peut **alimenter d’autres plugins** comme File Statistics ou Pattern Analysis
- Transition possible : les métriques CPU/mémoire peuvent déclencher alertes, ajuster d’autres plugins, etc.

## Plugins panel (dockables)

### Barchart (dans Data Inspector)

**Objectif :**  
Visualiser graphiquement des métriques ou statistiques directement depuis la **sélection active dans l’HexEditor**, de manière rapide et responsive.  

**Fonctionnalités :**  
- Affichage graphique en **barchart** ou histogramme.  
- Par défaut, **travaille uniquement sur la vue/selection en focus** dans l’HexEditor qui a le focus.  
- Doit pouvoir travaillé avec le plugin Pattern Analysis (sans en etre obligé)

  - Vue actuelle (default)  
  - Tout le fichier  
  - Bloc personnalisé

#20 UI ID Management pour Hot-Load / Plugin SDK

## 20.1. Clarification sur l’ID unique UI

Chaque **élément UI** (panel, menu, document tab, toolbar button) doit avoir un **ID unique** généré par le SDK pour éviter toute collision lors du **hot-load**.

### Pattern recommandé pour les IDs
- {PluginId}.{ElementType}.{ElementName}

**Exemples :**
- `WpfHexEditor.Plugins.Barchart.PanelMonitoring`
- `WpfHexEditor.Plugins.FileStatistics.MenuExport`

**Avantages :**
1. Garantie d’unicité globale.
2. Identification claire du plugin propriétaire.
3. Facilite le **hot-unload** en retirant tous les éléments liés au plugin.

---

##20.2. Intégration dans `IUIRegistry`

Le SDK expose les méthodes pour enregistrer et désenregistrer les éléments UI.

```csharp
public interface IUIRegistry
{
    string GenerateUIId(string pluginId, string elementType, string elementName);

    void RegisterPanel(string uiId, Panel panel, string pluginId);
    void RegisterMenuItem(string uiId, MenuItem menuItem, string pluginId);
    
    void UnregisterPanel(string uiId);
    void UnregisterMenuItem(string uiId);
    
    void UnregisterAllForPlugin(string pluginId);
}

Workflow :

1. Générer l’ID via GenerateUIId.
2. Vérifier collision (Exists(uiId)).
3. Enregistrer l’élément UI.
4. Au hot-unload, utiliser UnregisterAllForPlugin(pluginId) pour retirer tous les éléments du plugin.

##20.3. Vérification dans le manifest

Optionnel : déclarer tous les éléments UI dans le manifest pour une pré-validation avant runtime.

```JSON
"uiElements": [
  "WpfHexEditor.Plugins.Barchart.PanelMonitoring",
  "WpfHexEditor.Plugins.Barchart.MenuRefresh"
]

Avantages :
- Détection de collision avant chargement.
- Documentation explicite des éléments UI exposés par le plugin.

Voici un Mermaid diagram illustrant le workflow complet de génération d’ID UI, enregistrement, hot-load et hot-unload

flowchart TD
    A[Plugin démarre] --> B[SDK: GenerateUIId(pluginId, elementType, elementName)]
    B --> C{ID déjà existant ?}
    C -->|Oui| D[Log warning / suffix auto ou marquer Faulted]
    C -->|Non| E[RegisterPanel / RegisterMenuItem via IUIRegistry]
    E --> F[Élément UI visible dans IDE]
    F --> G{Hot-Unload demandé ?}
    G -->|Oui| H[UnregisterAllForPlugin(pluginId)]
    H --> I[Élément retiré de l’IDE]
    G -->|Non| F
    D --> F
    I --> J[Plugin terminé / IDE stable]

##20.4. Hot-Load sécurisé

Si un plugin tente de charger un UI ID déjà existant :
- Log warning / panel Output.
- Option 1 : auto-générer un suffixe (_1, _2).
- Option 2 : refuser le plugin et marquer comme "Faulted".

Objectif : aucun crash de l’IDE même avec plugin mal configuré.
Hot-unload : retirer tous les éléments via UnregisterAllForPlugin.

Résumé :
- Chaque élément UI a un ID unique.
- IDs générés par SDK suivant un pattern standard.
- Manifest peut déclarer les éléments UI pour pré-validation.
- Hot-load / unload entièrement sécurisés sans crash IDE.

- structures de données optimisées

## 32.6 Validation de performance

## 32.7 Objectif

# 35 BUS D’ÉVÉNEMENTS (IDE EVENT SYSTEM)

Afin de permettre la communication entre les différents composants de l’IDE sans créer de dépendances directes entre eux, le système doit utiliser un bus d’événements centralisé.

Ce mécanisme permet aux composants de publier et d’écouter des événements sans connaître les implémentations concrètes des autres composants.

## 35.1 Objectif du bus d’événements

Le bus d’événements permet :

- la communication entre composants
- la réduction des dépendances directes
- l’extensibilité du système
- l’intégration facile des plugins

Les composants peuvent publier des événements et d'autres composants peuvent s’y abonner.

## 35.2 Principe de fonctionnement

Le système repose sur deux opérations principales :

### Publication d’événement

Un composant peut publier un événement lorsqu’une action importante se produit.

Exemples :

- fichier ouvert
- fichier fermé
- solution ouverte
- plugin chargé
- erreur générée
- build terminé

### Abonnement aux événements

Les composants peuvent s’abonner à certains événements pour réagir lorsqu’ils se produisent.

Exemples :

- un panel peut se mettre à jour lorsqu’un fichier est ouvert
- un plugin peut analyser un fichier ouvert
- le terminal peut afficher un message

## 35.3 Architecture recommandée

Structure générale :

Event  
→ EventBus  
→ Publisher  
→ Subscribers

Le bus d’événements agit comme un intermédiaire entre les composants.

Les composants ne doivent pas se référencer directement entre eux lorsqu’un événement peut être utilisé.

## 35.4 Types d’événements

Le système peut inclure plusieurs catégories d’événements :

### Événements IDE

- solution ouverte
- solution fermée
- fichier ouvert
- fichier fermé
- éditeur activé

### Événements plugins

- plugin chargé
- plugin déchargé
- plugin initialisé

### Événements UI

- panel ouvert
- panel fermé
- panel activé

### Événements système

- erreur interne
- tâche terminée
- progression d’opération

## 35.5 Utilisation par les plugins

Le SDK doit permettre aux plugins :

- de publier des événements
- de s’abonner à des événements
- de réagir aux actions de l’IDE

Cela permet aux plugins d’interagir avec l’IDE sans dépendre directement de ses composants internes.

## 35.6 Bonnes pratiques

Claude doit privilégier l’utilisation du bus d’événements lorsque :

- plusieurs composants doivent être informés d’une action
- une fonctionnalité doit rester extensible
- une interaction indirecte est préférable à une dépendance directe

Claude doit éviter de créer des dépendances directes lorsque le bus d’événements est plus approprié.

## 35.7 Objectif

Ce système permet :

- une architecture modulaire
- une meilleure extensibilité
- une réduction des dépendances
- une intégration facilitée des plugins
- une meilleure évolutivité de l’IDE

ANNEXE: Mermaid complet “end-to-end”

Voici un Mermaid complet “end-to-end” pour le cycle complet d’un plugin WpfHexEditorIDE, depuis le build manifest jusqu’au hot-load et monitoring, incluant packaging et distribution.

flowchart TD
    %% 1. Développement
    A[Dev: Create Plugin using WpfHexEditor.SDK] --> B[dotnet build]
    B --> C[Generate Build Manifest (manifest.json)]
    C --> D{Manifest Valid?}
    D -->|No| E[Fail Build / Log Error]
    D -->|Yes| F[Build Artifact: DLL + Manifest]

    %% 2. Packaging
    F --> G[WhixPack / Packaging Tool]
    G --> H[Read Build Manifest]
    H --> I[Compute SHA-256 of DLL]
    I --> J[Inject Hash & File Size into Manifest]
    J --> K[Optionally Sign Package]
    K --> L[Add Marketplace / trustedPublisher metadata]
    L --> M[Generate Distribution Package (.whix)]
    
    %% 3. Installation via PluginHost
    M --> N[PluginHost: Discover Plugins Folder]
    N --> O[Read Distribution Manifest from .whix]
    O --> P{Manifest Valid / Integrity Check?}
    P -->|No| Q[Mark Plugin Faulted / Log Error]
    P -->|Yes| R[Check SDK & IDE Version Compatibility]
    R --> S{Compatible?}
    S -->|No| Q
    S -->|Yes| T[Select Isolation Mode: InProcess / Sandbox]
    T --> U[Load Assembly / Instantiate Plugin]
    
    %% 4. Initialization & UI Registration
    U --> V[SDK: GenerateUIId(pluginId, elementType, elementName)]
    V --> W{Collision ID?}
    W -->|Yes| X[Auto-suffix / Mark Faulted / Log Warning]
    W -->|No| Y[Register UI via IUIRegistry]
    Y --> Z[Plugin UI Visible in IDE]
    X --> Z

    %% 5. Focus & Monitoring
    Z --> AA[FocusContextService Notifies Plugin of Active Panel / Document]
    Z --> AB[PluginDiagnostics: CPU & Memory Monitor]
    AA --> Z
    AB --> Z

    %% 6. Hot-Load / Hot-Unload
    Z --> AC{Hot-Unload Requested?}
    AC -->|Yes| AD[UnregisterAllForPlugin(pluginId)]
    AD --> AE[Plugin UI Removed / IDE Stable]
    AC -->|No| Z
    Q --> AE
