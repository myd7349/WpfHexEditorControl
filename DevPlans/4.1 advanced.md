
# 40 SYSTÈME DE LOGS ET DIAGNOSTICS

Un IDE moderne nécessite un système centralisé de logs et diagnostics pour surveiller, déboguer et analyser le comportement du logiciel, y compris les plugins et les scripts exécutés via le terminal.

Claude ne doit pas créer de logs dispersés ou non standardisés.

## 40.1 Objectif

Le système de logs et diagnostics permet :

- de tracer les erreurs et exceptions
- d’analyser le comportement des fonctionnalités et plugins
- de surveiller les performances
- de fournir des informations pour le support et le debugging
- de centraliser les informations de diagnostic pour tous les composants

## 40.2 Architecture recommandée

- **LogService** : service central pour l’écriture et la lecture des logs
- **DiagnosticService** : collecte des métriques et informations de performance
- **LogSink** : abstraction pour les destinations (fichier, console, UI, serveur)

Structure simplifiée :

```
LogService
├─ LogInfo(message, source)
├─ LogWarning(message, source)
├─ LogError(message, exception, source)
├─ GetLogs(filter)
├─ ClearLogs()
```

```
DiagnosticService
├─ TrackEvent(name, properties)
├─ TrackPerformance(metric, value)
├─ GetMetrics()
```

## 40.3 Types de logs

1. **Logs IDE** : erreurs, avertissements, informations générales
2. **Logs Plugins** : erreurs et activités des plugins via SDK
3. **Logs Terminal / Scripts** : exécution et résultats des commandes
4. **Logs de performance** : temps d’exécution, consommation mémoire, accès disque

## 40.4 Intégration avec l’IDE

- Les logs doivent pouvoir s’afficher dans un **Output Panel** ou un **Error Panel**
- Les exceptions critiques doivent être remontées via le bus d’événements (#35) si nécessaire
- Le terminal doit pouvoir afficher certains logs pour les scripts ou commandes

## 40.5 Gestion des niveaux

Le système doit supporter différents niveaux de logs :

| Niveau | Description |
|--------|-------------|
| Info | Informations générales sur l’exécution |
| Warning | Avertissements sur un comportement inhabituel |
| Error | Erreurs nécessitant attention ou action |
| Debug | Informations détaillées pour le debugging |

Les niveaux doivent être configurables par l’utilisateur et filtrables.

## 40.6 Extensibilité pour les plugins

- Les plugins doivent pouvoir écrire dans le système de logs via le SDK
- Les plugins peuvent définir leur source ou catégorie
- Les logs des plugins doivent être séparés ou filtrables pour éviter le bruit

## 40.7 Bonnes pratiques

Claude doit respecter :

- centraliser tous les logs via LogService
- ne pas écrire directement dans la console ou des fichiers dispersés
- documenter les sources et catégories pour chaque log
- collecter des métriques pertinentes pour le diagnostic et la performance

## 40.8 Objectif

Ce système permet :

- une visibilité complète sur l’exécution de l’IDE
- un débogage facilité pour l’IDE et les plugins
- la collecte de métriques et la surveillance de la performance
- la maintenance et le support simplifiés

# 41 GESTION DES EXTENSIONS ET MARKETPLACE DE PLUGINS

Pour un IDE moderne et extensible, il est essentiel de fournir un système centralisé pour gérer les plugins et extensions, ainsi qu’une interface (marketplace) pour les découvrir, installer et mettre à jour.

Claude ne doit pas créer de plugins de manière dispersée ni sans passer par ce système.

## 41.1 Objectif

Le système d’extensions permet :

- d’installer et désinstaller facilement des plugins
- de mettre à jour les plugins
- de gérer les dépendances entre plugins
- de vérifier la compatibilité avec le SDK et l’IDE
- de fournir un mécanisme sécurisé pour l’exécution des plugins

## 41.2 Architecture recommandée

- **PluginManager** : service central pour l’installation, la désinstallation et la gestion des plugins
- **PluginRepository / Marketplace** : interface pour découvrir et télécharger des plugins
- **PluginLoader** : responsable du chargement dynamique des plugins au démarrage ou à la demande
- **VersioningService** : vérifie la compatibilité avec le SDK et l’IDE

Structure simplifiée :

```
PluginManager
├─ InstallPlugin(source)
├─ UninstallPlugin(pluginId)
├─ EnablePlugin(pluginId)
├─ DisablePlugin(pluginId)
├─ UpdatePlugin(pluginId)
├─ ListInstalledPlugins()
```

## 41.3 Gestion de la compatibilité

Le système doit :

- vérifier que le plugin cible la version correcte du SDK (#30)
- prévenir l’utilisateur en cas d’incompatibilité
- permettre le rollback en cas de mise à jour problématique
- isoler les plugins pour éviter que des erreurs affectent l’IDE

## 41.4 Marketplace / Repository

- Le Marketplace peut être local ou en ligne
- Les plugins doivent être signés ou vérifiés pour la sécurité
- Les métadonnées doivent inclure : version, auteur, dépendances, description, tags
- Les utilisateurs peuvent rechercher, filtrer et installer facilement les plugins

## 41.5 Cycle de vie des plugins

Le cycle de vie doit inclure :

1. Téléchargement / installation
2. Activation / désactivation
3. Chargement dynamique via PluginLoader
4. Communication via EventBus (#35) et Service Container (#36)
5. Désinstallation propre (suppression des fichiers, nettoyage des hooks et services)

## 41.6 Extensibilité

- Les plugins doivent pouvoir ajouter des panels (#33), des commandes (#37), des bindings (#38), et interagir avec le terminal (#34)
- Le SDK fournit les API nécessaires pour l’intégration sans accéder directement aux composants internes
- Le PluginManager doit gérer ces extensions et assurer la cohérence de l’IDE

## 41.7 Bonnes pratiques

Claude doit :

- centraliser la gestion via PluginManager
- vérifier la compatibilité et l’intégrité des plugins
- isoler les plugins pour éviter les crashs de l’IDE
- documenter le processus d’installation et de développement pour le SDK

## 41.8 Objectif

Ce système permet :

- un écosystème de plugins stable et extensible
- une intégration simple et sécurisée pour les utilisateurs
- une architecture cohérente pour les extensions
- un marché de plugins facilement exploitable et maintenable

# 42 SÉCURITÉ ET SANDBOXING DES PLUGINS

Pour garantir la stabilité et la sécurité de l’IDE, tous les plugins doivent être isolés et exécutés dans un environnement contrôlé.  
Claude ne doit pas permettre l’exécution de plugins avec accès direct non contrôlé aux composants critiques de l’IDE.

## 42.1 Objectif

Le système de sécurité des plugins permet :

- d’isoler les plugins pour éviter les crashes ou blocages de l’IDE
- de limiter l’accès aux ressources sensibles
- de protéger les données de l’utilisateur
- de contrôler l’exécution des scripts et commandes fournies par les plugins
- de sécuriser l’extension via le SDK et la marketplace (#41)

## 42.2 Principes de sandboxing

Chaque plugin doit être exécuté dans un environnement limité :

- accès restreint aux services exposés par le SDK (#30, #36)
- pas d’accès direct aux fichiers système ou à l’IDE sans autorisation
- communication via EventBus (#35) ou services contrôlés
- isolation mémoire pour éviter qu’un crash se propage

## 42.3 Contrôle des permissions

Le PluginManager doit vérifier et gérer les permissions pour chaque plugin :

- accès aux panels (#33)
- exécution de commandes (#37)
- accès aux fichiers et dossiers
- utilisation du terminal (#34)
- accès aux services internes (#36)

Les plugins doivent déclarer leurs permissions au moment de l’installation ou de l’activation.

## 42.4 Vérification de l’intégrité

Avant activation, le système doit :

- vérifier la signature numérique du plugin
- confirmer la compatibilité avec le SDK et l’IDE
- analyser les dépendances et éviter les conflits
- prévenir l’utilisateur en cas de risque potentiel

## 42.5 Isolation d’exécution

- Les plugins doivent être exécutés dans des AppDomains séparés ou équivalents
- Les exceptions doivent être capturées et loggées via LogService (#40)
- Les crashs d’un plugin ne doivent pas affecter l’IDE ou d’autres plugins
- Les threads ou tâches utilisés par les plugins doivent être contrôlés et limités

## 42.6 Surveillance et logging

- Toutes les actions critiques des plugins doivent être loggées
- Les erreurs, exceptions et violations de permissions doivent être reportées
- Le terminal peut afficher des messages de diagnostic pour les plugins

## 42.7 Bonnes pratiques

Claude doit :

- exiger la déclaration des permissions pour chaque plugin
- interdire tout accès direct aux composants internes non exposés
- isoler l’exécution pour limiter l’impact des crashs
- utiliser les services et EventBus pour les interactions
- assurer la compatibilité avec la sandbox pour les scripts du terminal

## 42.8 Objectif

Ce système permet :

- un IDE stable même avec des plugins tiers
- une sécurité renforcée pour l’utilisateur et les données
- une intégration sécurisée avec le SDK et la marketplace
- une architecture fiable et modulaire pour les extensions

# 43 MISE À JOUR AUTOMATIQUE DE L’IDE ET DES PLUGINS

Pour garantir que l’IDE reste sécurisé, stable et performant, il doit pouvoir mettre à jour automatiquement ses composants et les plugins via un mécanisme centralisé.  
Claude ne doit pas permettre de mises à jour dispersées ou non vérifiées.

## 43.1 Objectif

Le système de mise à jour permet :

- de maintenir l’IDE à jour avec les dernières fonctionnalités et corrections
- de mettre à jour les plugins installés via le marketplace (#41)
- de vérifier la compatibilité avant installation
- d’assurer la sécurité et l’intégrité du logiciel
- de faciliter le déploiement pour l’utilisateur

## 43.2 Architecture recommandée

- **UpdateService** : service central responsable des mises à jour
- **UpdateChecker** : vérifie la disponibilité des nouvelles versions
- **Downloader** : télécharge les mises à jour de manière sécurisée
- **Installer** : applique les mises à jour après vérification
- **RollbackManager** : permet de revenir à la version précédente en cas de problème

Structure simplifiée :

```
UpdateService
├─ CheckForUpdates(target: IDE | Plugin)
├─ DownloadUpdate(target)
├─ InstallUpdate(target)
├─ Rollback(target)
├─ NotifyUser(message)
```

## 43.3 Mise à jour de l’IDE

- L’UpdateService doit vérifier régulièrement les nouvelles versions
- Les mises à jour doivent être téléchargées et installées de façon sécurisée
- Les dépendances internes et SDK (#30) doivent être vérifiées
- Les crashs ou interruptions doivent déclencher un rollback sécurisé

## 43.4 Mise à jour des plugins

- Les plugins installés via le Marketplace (#41) doivent être mis à jour automatiquement si activés
- Les nouvelles versions doivent être vérifiées pour compatibilité et permissions
- Les plugins peuvent être mis à jour individuellement ou en lot

## 43.5 Sécurité et intégrité

- Les mises à jour doivent être signées numériquement
- Le système doit vérifier les signatures avant installation
- Les fichiers critiques ne doivent pas être remplacés directement sans sauvegarde
- Toute erreur doit être loggée via LogService (#40)

## 43.6 Notifications et contrôle utilisateur

- L’utilisateur doit être notifié des mises à jour disponibles
- L’utilisateur peut choisir d’installer immédiatement ou plus tard
- Les mises à jour peuvent être configurées en automatique ou manuelles

## 43.7 Bonnes pratiques

Claude doit :

- centraliser toutes les mises à jour via UpdateService
- vérifier la compatibilité des mises à jour avec l’IDE et les plugins
- assurer la possibilité de rollback
- utiliser les services et EventBus (#36, #35) pour notifier les composants
- logguer toutes les opérations pour diagnostic (#40)

## 43.8 Objectif

Ce système permet :

- un IDE et des plugins toujours à jour
- une expérience utilisateur sécurisée et stable
- une intégration cohérente avec le SDK et le Marketplace
- une maintenance simplifiée et centralisée

# 44 SYSTÈME DE DEBUGGING INTÉGRÉ

Un IDE moderne doit fournir un système de debugging intégré pour les projets, scripts et plugins, afin de faciliter le développement, le test et la maintenance.  
Claude ne doit pas créer de mécanismes de debugging isolés hors de ce système central.

## 44.1 Objectif

Le système de debugging intégré permet :

- d’exécuter et d’analyser des programmes ou scripts pas à pas
- d’inspecter les variables et l’état interne des composants
- de définir des points d’arrêt et des conditions
- de tracer l’exécution des plugins et scripts du terminal (#34)
- de centraliser les informations dans les panels (#33) et logs (#40)

## 44.2 Architecture recommandée

- **DebuggerService** : service central de contrôle du debugging
- **BreakpointsManager** : gestion des points d’arrêt
- **WatchWindow** : visualisation des variables et expressions
- **CallStackPanel** : affichage de la pile d’exécution
- **Integration avec Terminal et Panels** : pour afficher les résultats et messages

Structure simplifiée :

```
DebuggerService
├─ StartDebug(target)
├─ StopDebug()
├─ StepInto()
├─ StepOver()
├─ StepOut()
├─ SetBreakpoint(file, line)
├─ RemoveBreakpoint(file, line)
├─ Evaluate(expression)
```

## 44.3 Fonctionnalités principales

1. **Points d’arrêt** : lignes spécifiques, conditions ou exceptions
2. **Step execution** : StepInto, StepOver, StepOut
3. **Inspection** : variables locales, globales et objets complexes
4. **Call stack** : visualisation complète des appels en cours
5. **Logging et Output** : intégration avec OutputPanel et LogService (#40)
6. **Scripts et plugins** : support du debugging pour scripts exécutés via terminal (#34) et plugins (#41, #42)

## 44.4 Extensibilité pour les plugins

- Les plugins peuvent déclarer des points d’arrêt ou observer les événements du debugger
- Les plugins peuvent interagir avec DebuggerService via le SDK (#36)
- Les résultats du debugging des plugins doivent être isolés pour éviter de corrompre l’IDE

## 44.5 Intégration avec EventBus et Panels

- Tous les événements de debugging doivent passer par EventBus (#35)
- Les panels dédiés au debugging (#33) affichent l’état et les résultats
- Le terminal peut recevoir les messages de débogage et les exceptions

## 44.6 Bonnes pratiques

Claude doit respecter :

- centraliser toutes les opérations de debugging via DebuggerService
- isoler le debugging des plugins et scripts
- logguer les événements critiques pour diagnostic (#40)
- permettre la configuration des panels et affichages pour l’utilisateur

## 44.7 Objectif

Ce système permet :

- un débogage fiable pour l’IDE, les projets, scripts et plugins
- l’inspection détaillée de l’état interne
- l’intégration cohérente avec panels, terminal et logs
- une expérience utilisateur complète et productive
- un environnement sécurisé et extensible pour le développement

# 45 SYSTÈME DE TESTS INTÉGRÉS ET VALIDATION DES PLUGINS ET SCRIPTS

Pour assurer la stabilité, la sécurité et la compatibilité, l’IDE doit disposer d’un système de tests intégré permettant de valider les plugins, les scripts du terminal (#34) et les projets utilisateurs.  
Claude ne doit pas exécuter de plugins ou scripts sans validation préalable.

## 45.1 Objectif

Le système de tests et validation permet :

- de vérifier la compatibilité des plugins avec le SDK (#36, #41)
- de tester les scripts et commandes (#34, #37) avant exécution
- d’assurer la stabilité de l’IDE et la sécurité des données
- de fournir des rapports détaillés pour l’utilisateur et le développeur
- de faciliter l’intégration continue et les mises à jour (#43)

## 45.2 Architecture recommandée

- **TestRunnerService** : service central pour exécuter les tests
- **PluginTestSuite** : ensemble de tests automatisés pour les plugins
- **ScriptValidator** : vérification syntaxique et comportementale des scripts
- **ReportManager** : collecte et présentation des résultats de tests
- **Integration avec EventBus et LogService** : notifications et logging centralisé

Structure simplifiée :

```
TestRunnerService
├─ RunPluginTests(pluginId)
├─ RunScriptTests(scriptId)
├─ RunProjectTests(projectId)
├─ GetTestReport(target)
├─ ValidateScript(scriptCode)
```

## 45.3 Types de tests

1. **Tests fonctionnels** : vérifient que le plugin ou script remplit sa fonction attendue
2. **Tests de compatibilité** : vérifient la compatibilité avec le SDK et l’IDE (#30, #36)
3. **Tests de sécurité** : vérifient les permissions et sandboxing (#42)
4. **Tests de performance** : mesures de temps et ressources consommées
5. **Tests de stabilité** : s’assurent qu’aucun crash ou exception non gérée ne se produit

## 45.4 Extensibilité pour les plugins

- Les plugins peuvent fournir des suites de tests automatisés via le SDK
- Les tests peuvent être exécutés manuellement ou automatiquement à l’installation / mise à jour
- Les résultats sont isolés pour éviter l’impact sur l’IDE principal

## 45.5 Intégration avec panels et terminal

- Les rapports de tests peuvent s’afficher dans un **Test Panel** dédié (#33)
- Les résultats peuvent également être consultés via le terminal (#34)
- Les événements et erreurs doivent passer par EventBus (#35) et être loggués (#40)

## 45.6 Bonnes pratiques

Claude doit respecter :

- centraliser l’exécution des tests via TestRunnerService
- valider tous les plugins et scripts avant exécution
- fournir des rapports détaillés et exploitables
- intégrer le processus dans le workflow d’installation et mise à jour (#41, #43)
- assurer la compatibilité avec la sandboxing (#42)

## 45.7 Objectif

Ce système permet :

- une validation systématique des plugins et scripts
- la prévention des erreurs et crashs dans l’IDE
- une sécurité renforcée et un contrôle des permissions
- une meilleure expérience utilisateur et développeur
- l’intégration d’un workflow qualité pour tout l’écosystème de l’IDE

Le système de documentation et aide contextuelle permet :

- d’accéder rapidement aux guides et références pour l’IDE, les commandes (#37) et le SDK (#36)

```
DocumentationService
```

## 46.5 Extensibilité pour les plugins

- Les pages de documentation des plugins peuvent apparaître dans le système central
- La recherche et l’aide contextuelle doivent inclure les extensions des plugins

## 46.6 Bonnes pratiques

Claude doit :

- centraliser toute la documentation via DocumentationService
- maintenir la cohérence et l’accès contextuel
- permettre la mise à jour dynamique des contenus
- documenter clairement les API, commandes et points d’extension pour les plugins
- intégrer la documentation dans l’expérience utilisateur et développeur

## 46.7 Objectif

Ce système permet :

- un accès rapide et contextuel à l’information
- une meilleure productivité et compréhension de l’IDE
- une intégration complète avec commandes, panels, terminal et plugins
- une documentation maintenable et extensible
- une expérience cohérente pour utilisateurs et développeurs

# 47 SYSTÈME DE NOTIFICATIONS ET ALERTES

Pour un IDE moderne et réactif, un système centralisé de notifications et alertes permet de tenir l’utilisateur informé des événements importants, mises à jour, erreurs et résultats de scripts/plugins.  
Claude ne doit pas créer de notifications dispersées ou non standardisées.

## 47.1 Objectif

Le système de notifications permet :

- d’informer l’utilisateur en temps réel des événements critiques
- de centraliser toutes les alertes provenant des panels (#33), terminal (#34), commandes (#37), plugins (#41‑#42) et services (#36)
- de gérer la priorité et le type des notifications
- de fournir une interface cohérente pour afficher, filtrer et gérer les messages

## 47.2 Architecture recommandée

- **NotificationService** : service central responsable de la création, gestion et affichage des notifications
- **NotificationManager** : gère la file des notifications et leur priorité
- **Integration avec EventBus (#35)** : toutes les notifications passent par le bus d’événements
- **Panels et UI** : affichage dans des zones dédiées (ex. panneau notifications, toast, popup contextuel)

Structure simplifiée :

```
NotificationService
├─ SendNotification(type, message, source, priority)
├─ DismissNotification(notificationId)
├─ GetActiveNotifications()
├─ ClearAll()
```

## 47.3 Types de notifications

1. **Information** : messages généraux ou succès d’opérations
2. **Avertissement** : problèmes non critiques nécessitant l’attention
3. **Erreur** : erreurs critiques ou échec de commandes/plugins
4. **Mises à jour** : nouvelles versions de l’IDE ou des plugins (#43)
5. **Système** : alertes du terminal (#34), debugger (#44), tests (#45) ou performance (#40)

## 47.4 Gestion de la priorité

- Chaque notification peut avoir un niveau de priorité (Haute, Moyenne, Basse)
- Les notifications critiques peuvent interrompre ou attirer l’attention
- Les notifications informatives peuvent apparaître discrètement et disparaître automatiquement

## 47.5 Intégration avec les composants de l’IDE

- **Panels (#33)** : affichage centralisé des notifications
- **Terminal (#34)** : affichage des messages système et alertes de scripts
- **CommandBus (#37)** : certaines commandes déclenchent des notifications
- **Plugins (#41‑#42)** : peuvent générer des notifications via le SDK

## 47.6 Extensibilité pour les plugins

- Les plugins peuvent créer des notifications via NotificationService
- Les notifications doivent respecter le type et la priorité définis par l’IDE
- Les notifications des plugins doivent être filtrables et ne pas surcharger l’utilisateur

## 47.7 Bonnes pratiques

Claude doit :

- centraliser toutes les notifications via NotificationService
- utiliser EventBus (#35) pour propager les événements
- catégoriser et prioriser les notifications pour éviter la confusion
- documenter les types et sources disponibles pour le SDK
- permettre à l’utilisateur de filtrer ou configurer la présentation

## 47.8 Objectif

Ce système permet :

- une communication cohérente et efficace avec l’utilisateur
- le suivi des événements critiques de l’IDE et des plugins
- une interface claire pour les alertes et informations
- une expérience utilisateur plus productive et contrôlée

# 48 MONITORING ET ANALYTICS

Pour optimiser l’IDE, améliorer la performance et suivre l’utilisation des fonctionnalités et plugins, un système centralisé de monitoring et analytics est indispensable.  
Claude ne doit pas créer de collecte de données dispersée ou non sécurisée.

## 48.1 Objectif

Le système de monitoring et analytics permet :

- de suivre l’utilisation des fonctionnalités et commandes (#37‑#38)
- de mesurer les performances des panels (#33), terminal (#34) et plugins (#41‑#42)
- de détecter les erreurs, exceptions et comportements anormaux (#40)
- de générer des statistiques et rapports pour améliorer l’IDE et ses plugins
- de corréler les événements avec les notifications et alertes (#47)

## 48.2 Architecture recommandée

- **AnalyticsService** : service central pour collecter, agréger et stocker les données
- **EventTracker** : capture des événements via EventBus (#35)
- **PerformanceMonitor** : mesure de l’utilisation CPU, mémoire, temps d’exécution et latence
- **ReportManager** : consolidation et export des données pour analyse interne

Structure simplifiée :

```
AnalyticsService
├─ TrackEvent(eventName, properties)
├─ TrackPerformance(metricName, value)
├─ GetAnalyticsReport(filter)
├─ ExportReport(format)
```

## 48.3 Types de données collectées

1. **Usage des fonctionnalités** : commandes exécutées, panels utilisés, scripts lancés
2. **Performance** : temps d’exécution, consommation mémoire, latence
3. **Plugins** : utilisation et compatibilité des plugins (#41‑#42)
4. **Erreurs et exceptions** : issues capturées via LogService (#40)
5. **Interactions utilisateurs** : alertes et notifications consultées (#47)

## 48.4 Intégration avec l’IDE

- Les événements sont collectés via EventBus (#35)
- Les panels et terminal peuvent afficher les métriques locales (#33‑#34)
- Les plugins peuvent reporter des métriques anonymisées via le SDK (#36)
- Corrélation avec notifications et alertes (#47) pour analyser l’impact des événements

## 48.5 Sécurité et confidentialité

- Toutes les données collectées doivent être anonymisées
- Aucune information personnelle ou sensible ne doit être stockée ou transmise
- L’utilisateur peut désactiver la collecte de données à tout moment
- Les communications externes doivent être sécurisées et conformes aux réglementations (RGPD, etc.)

## 48.6 Extensibilité pour les plugins

- Les plugins peuvent générer des événements anonymisés via AnalyticsService
- Les métriques doivent être filtrées pour éviter la fuite d’information
- Les données collectées peuvent être utilisées pour améliorer les plugins et l’expérience utilisateur

## 48.7 Bonnes pratiques

Claude doit :

- centraliser toute la collecte via AnalyticsService
- sécuriser les données et respecter la confidentialité
- documenter les événements et métriques disponibles pour le SDK
- corréler les données avec les logs (#40) et notifications (#47) pour l’analyse
- utiliser les métriques pour planifier l’optimisation des fonctionnalités et plugins

## 48.8 Objectif

Ce système permet :

- un suivi complet de l’utilisation et performance de l’IDE
- la détection proactive des problèmes et erreurs
- l’optimisation continue de l’expérience utilisateur et développeur
- l’amélioration des plugins et de l’écosystème
- une base fiable pour les décisions de développement et les mises à jour (#43)
